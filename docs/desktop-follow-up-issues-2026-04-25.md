# 桌面端后续问题记录

> 日期：2026-04-25
> 范围：`apps/desktop`、`apps/local-web`、`apps/cli`

当前桌面端已经可以继续推进发布和分发，但仍有一些后续问题需要集中跟踪，避免问题散落在聊天记录或临时说明中。

## DESKTOP-001：桌面端安装包分发源需要重新规划

- **状态**：待规划
- **现象**：当前桌面端安装包放在 Gitee 上，但 Gitee 对 Release 附件、仓库容量和历史大文件都有配额限制。后续版本增多后，容易再次触发限制。
- **影响**：安装包上传、下载和版本发布可能不稳定，尤其是历史 Release 附件累积后更容易超过限制。
- **建议方向**：
  - Gitee 保留为国内入口或备用下载源。
  - GitHub Releases 可作为公开安装包主发布源或海外源。
  - Cloudflare R2 可作为备用直链或自动更新源，但不应作为国内唯一下载源。
  - Netlify 更适合放下载页、官网或说明页，不建议作为桌面端安装包主仓库。

## DESKTOP-002：Gitee 容量限制需要前置规避

- **状态**：待规划
- **现象**：即使单个安装包看起来不大，也可能因为 Gitee 的单文件、单仓库附件总容量、Git 仓库容量或历史提交体积触发限制。
- **影响**：Release 附件上传失败、Git push 失败，或删除文件后仓库仍然处于超限状态。
- **建议方向**：
  - 避免将安装包直接提交到 Git 仓库。
  - 定期清理旧 Release 附件，或迁移历史安装包。
  - 如果仓库历史已经包含大文件，需要单独安排仓库瘦身。

## DESKTOP-003：Cloudflare R2 国内访问稳定性不能作为唯一保障

- **状态**：待验证
- **现象**：普通 R2 免费或个人用法主要走 Cloudflare 全球网络，不等同于使用中国大陆节点。国内访问会受到地区、运营商和时间段影响。
- **影响**：国内用户下载桌面端安装包时，可能出现速度慢、连接不稳定或偶发无法访问。
- **建议方向**：
  - R2 作为海外或备用源使用。
  - 国内下载建议保留 Gitee，后续可接入国内对象存储或 CDN。
  - 下载页提供多个镜像链接，降低单一来源不可用的风险。

## DESKTOP-004：桌面端设置入口不可点击

- **状态**：待分析 / 待修复
- **现象**：桌面端的“设置”入口当前无法点击进入。
- **影响**：用户无法进入设置页调整桌面端配置，影响源配置、缓存策略、关闭行为等功能使用。
- **建议方向**：
  - 检查顶部设置按钮或入口的点击事件是否绑定正确。
  - 检查 hash 路由是否能正常切换到 `#settings`。
  - 检查 Tauri 环境下是否存在拖拽区域、遮罩层或禁用状态拦截点击。
  - 增加一次桌面端手动验证，确认普通浏览器和 Tauri 窗口中都能进入设置页。

## DESKTOP-005：已安装页顶部出现“技能名称无效”错误

- **状态**：待分析 / 待修复
- **现象**：桌面端进入“已安装”页面后，技能列表可以正常显示，但页面顶部出现红色错误横幅，文案为“技能名称无效”。
- **截图线索**：
  - 当前页面为“已安装”。
  - 列表中已经展示 `algorithmic-art` 等已安装技能。
  - 错误横幅出现在列表上方，不像是整个已安装列表加载失败。
- **初步判断**：
  - 更像是某一次详情、编辑、导出、复制包或目标链接操作触发了 `INVALID_SKILL_NAME`，但全局错误状态没有在页面切换或已安装列表成功加载后清理。
  - 也可能是已安装技能扫描时，有某个本地目录或元数据解析出了空名称、中文名称、带下划线名称、带作用域前缀名称等非法 skill name，后续操作把这个名称传给了 API。
  - 当前 `validateSkillName` 只允许小写字母、数字和短横线，且不能以短横线开头或结尾，因此空字符串、下划线、中文、空格、`@scope/name` 形式都会触发该错误。
- **建议方向**：
  - 检查 `apps/local-web/src/App.tsx` 中全局 `error` 状态是否在进入“已安装”页或 `loadInstalled()` 成功后清空。
  - 检查 `fetchInstalled()` / Tauri installed bootstrap 返回的 `items` 是否存在 `name` 为空或不符合 `validateSkillName` 的条目。
  - 对已安装列表做防御：非法名称条目应显示明确的“本地技能目录名称不合法”提示，而不是统一弹出“技能名称无效”。
  - 对已安装页操作入口做前置校验，避免把非法名称继续传给编辑、导出、复制、删除和链接目标接口。

## DESKTOP-006：技能详情页提示“技能元信息丢失”

- **状态**：待分析 / 待修复
- **现象**：进入技能详情页后，页面顶部出现“技能元信息丢失”的提示；但左侧文件树已选中 `commit-helper`，右侧 `SKILL.md` 正文内容可以正常渲染。
- **截图线索**：
  - 当前页面为技能详情页，面包屑已经进入 `commit-helper`。
  - 详情正文标题 `Commit Helper`、说明文本和代码块均可显示。
  - 顶部翻译工具栏可显示，并显示“翻译中 0/15”。
  - 错误位置位于详情正文上方，说明文件内容本身不是空的。
- **初步判断**：
  - 这不是单纯的文件读取失败；`SKILL.md` 正文还在，但顶部 frontmatter 元信息大概率在“美化格式”时被去掉了。
  - 当前标准 skill 依赖 `SKILL.md` 顶部 frontmatter 作为主要元信息来源；如果这段被删除，系统只能读取正文标题和内容，无法解析 `name`、`description`、`version`、`author`、`tags` 等结构化字段。
  - 独立详情页的数据流仍需检查：`SkillDetailView` 通过 `fetchSkillBrowserBundle()` 加载文件树和初始内容，该 bundle 本身可能不包含完整 metadata。
- **建议方向**：
  - 批量检查被美化过的 `SKILL.md`，确认哪些文件缺少顶部 `--- ... ---` frontmatter。
  - 从目录名、正文一级标题和旧版本记录中恢复必要字段，至少补回 `name`、`description`、`version`、`author`、`tags`。
  - 调整后续美化或格式化流程：保留 frontmatter 原样，或只格式化 frontmatter 之后的正文。
  - 检查独立详情页是否需要同时调用 `fetchSkillDetail(skillName, source)`，或扩展 `SkillBrowserBundle` 返回 metadata，避免页面只知道正文内容。
  - 如果 frontmatter 缺失，应使用目录名作为 `name` fallback，并将 `metadataSource` 标为 `unknown`，不要直接显示“技能元信息丢失”。

## DESKTOP-007：整体翻译过慢且翻译结果没有稳定复用

- **状态**：待分析 / 待重构
- **现象**：技能详情页整体翻译速度过慢；已经翻译过的内容没有稳定记录下来，第二次进入同一技能或同一文件时仍然需要继续翻译。
- **当前实现观察**：
  - `SkillDetailView` 中存在内存缓存 `translateMemoryCache` 和 `localStorage` 缓存，缓存 key 由文本块 fingerprint 和目标语言组成。
  - 每次进入新内容时，组件会重置 `mode`、`translatedByKey`、`failedByKey`，再由 `ensureTranslations()` 逐块读取缓存或重新翻译。
  - 翻译任务由 `collectMarkdownTranslationTasks()` 拆成多个 markdown 节点，当前并发数为 `TRANSLATE_CONCURRENCY = 4`。
  - 缓存是文本块级别的，不是文件级别或技能级别的完整翻译快照。
- **初步判断**：
  - 慢的主要原因是逐段翻译：一个 `SKILL.md` 可能拆成十几到几十个任务，每个任务都要单独请求翻译接口。
  - 复用不稳定的原因可能包括：`localStorage` 在桌面端 WebView 中没有稳定持久化、缓存写入被 quota 或异常吞掉、文本被美化或格式化后 fingerprint 改变，或只缓存单块而没有保存整篇完成状态。
  - 当前 UI 状态每次进入都会清空，需要重新扫描所有任务并逐个查缓存；即使缓存命中，也会有明显的“重新计算 / 重新进入翻译流程”的体验。
- **建议方向**：
  - 将缓存从前端临时 `localStorage` 升级为桌面端或服务端持久缓存，例如按 `source + skillName + filePath + contentHash + targetLang + provider/model` 存储完整文件翻译结果。
  - 增加“整篇批量翻译”路径：优先把同一文件的多个文本块合并成批量请求，返回后再映射回 markdown 节点，减少请求次数。
  - 保存文件级完成状态和部分完成状态；二次进入时先同步恢复已翻译内容，再只补翻缺失块。
  - 缓存 key 需要包含内容 hash。当 `SKILL.md` 内容没有变化时，应直接复用整篇翻译。
  - 在翻译工具栏展示缓存命中状态，例如“已缓存”“恢复翻译结果”“需补翻 3 段”，避免用户误以为每次都从零开始。
  - 对失败块单独重试，不应导致已完成块重新请求。

## 后续验证

- 桌面端点击“设置”后应进入设置页。
- 已安装页不应在列表正常展示时残留“技能名称无效”错误横幅。
- 技能详情页在正文正常显示时，不应再提示“技能元信息丢失”，并应展示可降级的版本、作者、来源和 metadata 来源。
- 翻译结果应持久保存；同一技能同一文件第二次进入时，不应从零重新翻译。
- 下载页或发布说明应提供清晰的多源下载方案。
- 新版本发布时不应依赖单一托管平台。
