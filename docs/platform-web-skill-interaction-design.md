# Platform Web 技能市场交互设计说明

## 设计目标

本文档用于梳理 `platform-web` 中“技能市场、技能详情、评价、技能目录”四个区域的合理交互关系。目标不是增加页面数量，而是建立清晰的用户路径，让用户始终知道：

- 我从哪里进入当前页面。
- 我现在处于哪个技能和哪个模块。
- 当前操作完成后下一步应该去哪里。

整体设计应围绕一个闭环展开：

```text
技能市场 -> 技能详情 -> 评价 / 技能目录 -> 返回技能详情 -> 返回来源页
```

## 信息架构

“技能市场、评价、技能详情、技能目录”不应被设计成四个完全平级的页面。更合理的关系是：

```text
技能市场
  -> 技能详情
       -> 概览
       -> 安装与使用
       -> 评价
       -> 技能目录
```

其中：

- 技能市场：发现入口，负责搜索、筛选、比较和进入详情。
- 技能详情：决策中心，负责让用户判断是否安装、信任、评价或维护该技能。
- 评价：详情页中的信任模块，不建议作为普通用户的主导航入口。
- 技能目录：详情页下的高级操作区，用于查看或维护技能包文件。

## 技能市场

### 页面职责

技能市场应承担“快速发现”和“快速比较”的职责。它像一个技能货架，不应承担过多维护操作。

### 推荐展示信息

每个技能条目建议展示：

- 技能名称
- 简短描述
- 分类
- 标签
- 来源
- 作者
- 评分
- 评价数
- 安装量
- 状态，例如已验证、待审核、新发布
- 更新时间
- 主要入口：查看详情

### 推荐交互

- 点击整行进入技能详情页。
- 搜索、分类、来源、排序组合筛选时，应明显展示结果数量。
- 从详情页返回时，应保留搜索条件、筛选条件和滚动位置。
- 空状态应提供“清空筛选”或“返回全部技能”的操作。
- 市场列表中不建议直接放“编辑目录”“写评价”等深层操作。

## 技能详情

### 页面职责

技能详情是用户决策的中心。它需要回答：

- 这个技能解决什么问题？
- 是否值得安装？
- 是否可信？
- 有哪些评价？
- 是否可以查看或维护它的目录内容？

### 推荐页面结构

```text
返回来源页

技能名称 + 状态
技能描述
标签 / 来源 / 作者 / 版本

评分 / 安装量 / 评价数 / 更新时间

主要操作：
复制安装命令
查看评价
进入技能目录

技能说明
适用场景
安装与使用
评价摘要
技能目录入口
```

### 操作优先级

普通用户场景下，操作优先级建议为：

1. 复制安装命令或安装使用。
2. 查看评价。
3. 进入技能目录。

维护者或管理员场景下，可以提高“进入技能目录”的权重，但仍应保留“安装与使用”的信息，避免详情页变成纯编辑入口。

### 返回逻辑

详情页的返回按钮不应固定返回技能市场，而应返回用户来源：

```text
技能市场 -> 技能详情 -> 返回技能市场
我的技能包 -> 技能详情 -> 返回我的技能包
```

按钮文案也应根据来源变化：

- 从技能市场进入：返回技能市场
- 从我的技能包进入：返回我的技能包

## 评价

### 页面职责

评价用于帮助用户建立信任，也用于帮助维护者发现问题。普通用户侧的评价不建议设计成独立主导航模块。

### 推荐承载方式

第一阶段建议将评价放在技能详情页内，或以详情页中的弹窗、抽屉形式出现：

```text
技能详情
  -> 评价摘要
       -> 平均分
       -> 评价标签
       -> 最近评价
       -> 写评价
       -> 查看全部评价
```

评价很多时，再扩展为“查看全部评价”的二级视图。

### 评价表单

评价表单不建议默认完整展开。建议先展示“写评价”入口，点击后展开：

- 评分
- 标签，例如安装体验、文档清晰、效果优秀、稳定性、需要改进
- 评价内容
- 是否匿名
- 联系方式，非匿名时再显示

### 提交后反馈

提交成功后，不应只显示一行文案。建议：

- 刷新当前技能的评价摘要。
- 将新评价展示在评价列表中，或给出“评价已提交”的确认状态。
- 提供“继续浏览技能”或“关闭评价”的清晰出口。

### 与评价中心的区别

需要区分两个概念：

- 技能详情中的评价：面向普通用户，用于表达体验和辅助决策。
- 评价中心：面向管理员，用于审核、处理、归档和运营反馈。

普通用户不应被引导理解“评价中心”这个管理模块。

## 技能目录

### 页面职责

技能目录是高级操作区，用于查看技能包结构和维护文件内容。它不应直接暴露为技能市场的主操作，而应从技能详情进入。

### 推荐入口

在技能详情页中提供轻量说明：

```text
技能目录
查看 SKILL.md、meta.json、示例文件和资源内容。维护者可以编辑可写文件。

进入技能目录
```

### 推荐布局

```text
顶部：
返回技能详情 / 技能名称 / 当前路径 / 保存状态

主体：
左侧：文件树
中间：文件内容或编辑器
右侧：文件信息、校验结果、操作记录
```

### 推荐交互

- 默认打开 `SKILL.md`，避免进入后出现空白状态。
- 文件树点击文件后，中间区域即时切换内容。
- 可编辑文件和只读文件应明显区分。
- 修改后顶部显示“未保存”状态。
- 保存成功后显示明确反馈。
- 文件加载失败时提供“重新加载”和“返回详情”。
- 离开目录、返回详情、切换文件或点击侧边栏时，如果有未保存内容，应提示用户保存或放弃。

### 返回逻辑

目录页应始终返回技能详情，而不是直接返回技能市场：

```text
技能市场 -> 技能详情 -> 技能目录 -> 返回技能详情 -> 返回技能市场
我的技能包 -> 技能详情 -> 技能目录 -> 返回技能详情 -> 返回我的技能包
```

## 路由与状态规则

### 推荐路由关系

```text
/market
/skills/:skillId
/skills/:skillId/files
/mine
```

### 来源状态

进入技能详情时应记录来源：

```text
from = market | mine
```

来源可用于：

- 控制侧边栏高亮。
- 控制返回按钮文案。
- 控制返回目标。
- 保留市场筛选和滚动位置。

### 侧边栏高亮

深层页面也应有归属关系：

- `/skills/:skillId` 默认归属技能市场。
- `/skills/:skillId/files` 默认归属技能市场。
- 如果从我的技能包进入，则归属我的技能包。

这样用户进入详情或目录后，仍能理解当前页面属于哪个业务模块。

### 异常路径

不建议将所有未知路径静默重定向到技能市场。

推荐处理方式：

- 技能不存在或已下架：展示“技能不存在或已下架”，提供“返回技能市场”。
- 无权限访问：展示“当前账号无权限访问该页面”，提供返回入口。
- 未知路径：展示 404 页面，而不是直接跳转。

## 关键用户路径

### 从技能市场浏览技能

```text
进入技能市场
搜索或筛选技能
点击技能行
进入技能详情
查看说明、评分、评价和安装命令
复制安装命令或进入技能目录
返回技能市场，保留筛选条件和滚动位置
```

### 从我的技能包维护技能

```text
进入我的技能包
点击自己维护的技能
进入技能详情
进入技能目录
编辑文件
保存
返回技能详情
返回我的技能包
```

### 评价技能

```text
进入技能详情
查看评价摘要
点击写评价
填写评分、标签和内容
提交评价
刷新评价摘要
关闭评价区域或继续浏览详情
```

## 优先级建议

### P0

- 技能 ID 不存在时，不要自动展示第一个技能，应展示明确的不存在状态。
- 目录页存在未保存内容时，离开前必须提示。

### P1

- 详情页返回逻辑应尊重来源页。
- `/skills/*` 深层页面应有侧边栏归属高亮。
- 上传或创建技能后，应提供进入“我的技能包”或“技能详情”的明确下一步。

### P2

- 详情页增加面包屑或来源提示。
- 评价从弹窗逐步演进为详情页内的完整评价区。
- 404、无权限、加载失败等异常状态单独设计。

## 总结

这组页面的核心不是把功能全部露出来，而是让用户路径稳定：

```text
市场负责发现
详情负责判断
评价负责信任
目录负责维护
```

只要这四个职责边界清楚，页面跳转和操作优先级就会更自然，用户也不容易在市场、详情、评价和目录之间迷失。
