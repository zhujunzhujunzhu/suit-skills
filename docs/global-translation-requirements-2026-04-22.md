# 全局悬浮翻译需求整理

日期：2026-04-22

## 1. 需求背景

当前页面已经有“内容翻译”能力，但它本质上还是一个局部能力：

- 前端入口是 `web/src/App.tsx` 里的 `TranslateMarkdownView`
- 后端能力是 `src/lib/web/api.ts` 里的 `/api/translate`
- 设置页可以配置翻译 provider（OpenAI 兼容 API / 本地 CLI）

现有方案更像是“对某一段 Markdown 文本做一次整块翻译”，而不是“对当前页面内容进行全局翻译”。

你现在想要的是一个更靠近浏览器翻译插件的体验：

- 页面上有一个全局悬浮的翻译控制区
- 自动判断当前内容是不是英文
- 如果是英文，可以直接触发翻译
- 翻译后既可以替换原文，也可以在原文下方增加一行译文，形成双语对照

## 2. 当前实现现状

基于现有代码，当前翻译功能的实际边界如下：

### 2.1 已有能力

- 可以在设置页配置翻译服务
- 可以调用统一的 `/api/translate` 接口翻译文本
- 翻译结果会按 `localStorage` key 做缓存
- `SKILL.md` 详情区支持一键切换“原文 / 中文”
- 文件浏览器中，只有 `.md` 文件预览区复用了这套翻译逻辑

### 2.2 当前限制

- 不是全局能力，只挂在两个局部区域：
  - Skill 详情页的 Markdown 区域
  - 文件浏览器中的 `.md` 文件预览区域
- 翻译单位是“整段 Markdown 字符串”，不是页面中的多个内容块
- 当前交互只有二选一切换：
  - 显示原文
  - 显示译文
- 不支持双语对照显示
- 不做语言识别，只要用户点了就发起翻译
- 不区分页面中的内容类型，当前只覆盖 Markdown，不覆盖：
  - 列表描述
  - 普通文本区块
  - 弹层内容
  - 其他非 Markdown 的详情内容
- 当前缓存粒度偏文件级，适合“翻译一个文件”，不适合“翻译当前页面多个块”

## 3. 这次需求的目标定义

这次需求建议定义为：

“把现有的单 Markdown 翻译能力，升级为页面级的全局翻译能力，并提供统一的悬浮控制入口与双语显示模式。”

## 4. 功能目标

### 4.1 全局悬浮翻译入口

页面右侧或右下角增加一个全局悬浮翻译面板，至少包含：

- 当前状态展示
  - 未检测
  - 可翻译
  - 翻译中
  - 已翻译
  - 翻译失败
- 翻译模式切换
  - 替换原文
  - 双语对照
- 操作按钮
  - 立即翻译
  - 恢复原文

这个入口应当独立于具体 Markdown 组件存在，而不是继续绑死在某个文件内容区域上。

### 4.2 页面内容语言识别

系统需要先判断当前页面可翻译内容是否以英文为主。

建议规则：

- 如果页面主要内容为英文，则显示“可翻译”
- 如果页面主要内容为中文，则默认不触发翻译
- 如果页面内容混合，则只翻译被判定为英文的内容块

这里的“判断内容”应面向页面内容区，而不是整个 DOM。

### 4.3 翻译模式

需要支持两种模式：

#### 模式 A：替换原文

- 将原有英文内容直接替换为中文译文
- 用户可以点击“恢复原文”回退

#### 模式 B：双语对照

- 保留原英文内容
- 在原内容下方插入一行或一个块级区域展示中文译文
- 视觉上要明确区分“原文”和“译文”

双语模式更适合文档阅读、技能说明、教程类内容。

### 4.4 翻译范围

建议这次需求明确为“页面内容级”，覆盖当前页面中的可阅读文本块，而不只是一份 md 文件。

建议优先覆盖：

- Skill 详情正文
- 文件浏览器中的文本内容
- 页面中的描述类文本
- 列表中的英文说明文案
- 弹层或详情面板中的正文文本

建议默认不翻译：

- 代码块
- 行内代码
- 文件路径
- URL
- 命令行片段
- 纯符号或结构性字段
- 输入框中的用户编辑内容

## 5. 推荐的翻译单位

为了支持“替换原文”和“双语对照”，翻译单位不应再是整份 Markdown 文本，而应改成“页面中的文本块”。

建议以块级内容为最小单位，例如：

- 标题 `h1-h6`
- 段落 `p`
- 列表项 `li`
- 引用块 `blockquote`
- 表格单元格
- 描述性 `div/span` 文本块

这样做的好处：

- 可以只翻译英文块
- 可以局部失败，不影响整页
- 更容易做双语对照
- 更容易做缓存复用

## 6. 推荐交互方案

### 6.1 初始状态

- 页面加载后，悬浮面板扫描当前页面可翻译节点
- 如果检测到英文内容，则按钮可点击
- 如果未检测到英文内容，则按钮置灰，并提示“当前页无可翻译英文内容”

### 6.2 触发翻译

- 用户点击“立即翻译”
- 系统收集当前页所有待翻译文本块
- 对英文块进行批量翻译
- 翻译完成后按当前模式渲染

### 6.3 模式切换

- 已翻译后，用户可以在“替换原文”和“双语对照”之间切换
- 如果译文已缓存，模式切换不应再次请求翻译接口

### 6.4 恢复原文

- 一键移除所有译文渲染
- 页面恢复到原始内容

## 7. 推荐技术方案

### 7.1 不建议直接做纯 DOM 暴力替换

如果只是写一个“全局悬浮按钮 + 遍历 DOM + 直接改 innerText”的方案，短期能看到效果，但会有明显问题：

- React 重新渲染后译文容易丢失
- 组件切换后状态难维护
- 难以和缓存、模式切换、局部刷新协同
- 对代码块、图标、按钮、输入框的误伤概率高

所以，“全局悬浮”应该是全局控制入口，不代表底层一定要用原始 DOM 篡改。

### 7.2 推荐方案：全局控制 + 块级注册

更适合当前项目的方案是：

- 在页面根部新增一个全局 `TranslationProvider` 或全局翻译状态层
- 增加一个 `GlobalTranslationFloatingPanel` 组件作为悬浮控制区
- 各页面内容区把“可翻译文本块”注册到这个全局状态中
- 翻译结果按块缓存，再由渲染层决定：
  - 替换原文
  - 原文下追加译文

这样既能实现“全局控制”，又能保持 React 渲染的一致性。

### 7.3 后端接口建议升级

现有接口是：

- `POST /api/translate`
- 入参：单个 `text`
- 出参：单个 `translated`

为了支持页面级翻译，建议扩展成批量接口，或者新增一个批量接口：

- `POST /api/translate/batch`

建议入参结构：

```json
{
  "items": [
    {
      "id": "block-1",
      "text": "Install dependencies first.",
      "sourceLang": "en"
    },
    {
      "id": "block-2",
      "text": "Run the server locally.",
      "sourceLang": "en"
    }
  ],
  "targetLang": "zh-CN"
}
```

建议出参结构：

```json
{
  "items": [
    {
      "id": "block-1",
      "translated": "请先安装依赖。"
    },
    {
      "id": "block-2",
      "translated": "在本地运行服务。"
    }
  ],
  "provider": "openai"
}
```

这样前端更容易按块渲染、缓存、局部重试。

## 8. 缓存建议

当前缓存是文件级 key，后续建议改成“页面块级缓存”。

建议缓存 key 组成：

- 页面标识
- 内容块 id
- 原文 hash
- 目标语言

示例：

```text
translate:page:skill-detail:block-12:en:hash
```

这样可以避免：

- 页面切换后缓存串用
- 原文变化后旧译文被误复用
- 双语模式和替换模式重复请求

## 9. 性能优化补充需求

这一轮补充需求的重点，是把“能翻译”升级为“翻译速度可接受、用户能感知进度、已翻译内容可复用”。

### 9.1 当前性能瓶颈分析

结合现有实现，当前慢主要有几个原因：

- 当前前端是整段内容一次性调用 `translateText(markdown)`，必须等整段翻译完成后才能渲染
- 当前后端接口只支持单个 `text`，不支持批量块级翻译
- 当前翻译流程没有流式返回，前端只能一直等待最终结果
- 当前缓存更接近“页面实例缓存”，不是“内容指纹缓存”
- 当前没有请求去重机制，同一内容在不同页面或不同组件中可能重复翻译
- 如果使用本地 CLI 作为 provider，每次调用都要启动一个子进程，也会带来额外延迟

### 9.2 性能优化目标

补充后的目标建议明确为：

- 首屏不要因为整页翻译而长时间卡住
- 用户点击翻译后，尽快看到部分结果，而不是只能等全部结束
- 相同内容第二次出现时，优先命中本地缓存，避免再次请求
- 页面内多个块的翻译任务可以并发执行，但不能无限并发
- 当某些块翻译失败时，不影响其他块先显示结果

### 9.3 并发翻译需求

可以引入“受控并发”机制，而不是串行一个一个翻译。

建议要求如下：

- 页面内容切块后，多个英文块可以并发翻译
- 前端或后端需要设置并发上限，例如 3 到 6 个并发任务
- 并发上限应可配置，避免触发 provider 限流
- 同一原文内容如果在同一轮任务中重复出现，只发起一次翻译请求，其余块复用结果
- 某个块失败时，只标记该块失败，不阻塞整页其他块

建议不要做“无限并发”，否则容易带来：

- OpenAI 或兼容接口限流
- CLI 进程数量过多
- 浏览器主线程压力增大
- 页面切换时取消和回收困难

### 9.4 流式渲染需求

为了改善体感速度，建议支持“结果一出来就逐块展示”，而不是整页等待。

建议需求如下：

- 翻译任务启动后，页面立即进入“翻译中”状态
- 哪个文本块先返回，就先渲染哪个块的译文
- 双语模式下，可先显示原文，再逐块补上译文
- 替换模式下，可先保留原文占位，块翻译完成后再替换
- 悬浮面板显示整体进度，例如：
  - 总块数
  - 已完成块数
  - 失败块数

需要注意：

- 这里的“流式渲染”不一定要求底层 API 真正使用 SSE
- 第一版可以先做“块级并发 + 逐块回填渲染”
- 如果后续 provider 层支持流式输出，再升级为真正的服务端流式传输

### 9.5 本地缓存补充需求

“翻译过的内容直接存储在本地，下次遇到相同的内容直接复用”应当明确成正式需求。

建议缓存规则如下：

- 缓存粒度为文本块，不再只是整份文档
- 缓存 key 基于内容指纹，而不是页面位置
- 同一段英文内容，无论出现在哪个页面，只要内容一致，都可以直接复用译文
- 缓存值至少包含：
  - 原文
  - 原文 hash
  - 目标语言
  - 译文
  - provider
  - model
  - 创建时间
- 当原文内容变化时，旧缓存自动失效

推荐 key 示例：

```text
translate:block:sha256(<sourceText>):zh-CN:<provider>:<model>
```

这样做的意义是：

- 真正实现“相同内容秒开”
- 页面切换时仍可复用
- 不依赖块 id 或页面结构
- 为后续导出缓存、清理缓存提供基础

### 9.6 两级缓存建议

为了进一步提速，建议设计成两级缓存：

- 一级缓存：内存缓存
  - 作用：当前页面会话内秒级复用
  - 适合去重、避免同轮重复请求
- 二级缓存：`localStorage` 或本地持久缓存
  - 作用：跨页面、跨刷新复用
  - 适合保存稳定的翻译结果

建议优先读取顺序：

1. 先查内存缓存
2. 未命中再查本地持久缓存
3. 仍未命中才发起翻译请求

### 9.7 预热与懒加载策略

为了进一步优化体验，可以增加“预热”和“懒翻译”策略。

建议如下：

- 页面初次加载时只做语言检测，不立刻整页翻译
- 用户点击翻译后，优先翻译当前可视区域的文本块
- 非可视区域内容可以延后翻译
- 如果某个页面用户频繁访问，可以在空闲时预热缓存
- 列表页只在卡片进入视口或被展开时再翻译描述

这样可以避免：

- 一进页面就发起过多请求
- 对用户根本没看到的内容做无效翻译

### 9.8 去重与取消机制

性能优化不只是“更快发”，还包括“少做无效工作”。

建议补充以下能力：

- 相同内容的并发请求自动合并
- 页面切换、组件卸载或用户点击“恢复原文”时，未完成任务可取消
- 新一轮翻译任务开始时，应能中止上一轮无效任务
- 已完成块保留结果，未完成块停止继续消耗资源

### 9.9 错误与降级策略

为了保证性能优化方案更稳，建议同时定义降级策略：

- 如果批量接口失败，可回退到单块重试
- 如果流式模式不可用，可回退到逐块普通请求
- 如果 provider 限流，可自动降低并发度
- 如果本地缓存不可用，仍应允许在线翻译继续工作
- 如果某个块超时，显示原文并标记“翻译失败”，不拖住整页

### 9.10 建议新增的验收标准

在原有验收标准基础上，补充以下性能相关标准：

1. 页面翻译任务支持受控并发，而不是严格串行。
2. 用户点击翻译后，可以逐块看到结果回填，不必等待整页全部完成。
3. 相同英文内容在第二次出现时，优先命中本地缓存，不重复请求翻译服务。
4. 同一轮翻译中，相同内容不会重复发起多个请求。
5. 页面切换或取消翻译时，未完成任务能够停止，不继续浪费资源。
6. 某些块翻译失败时，其他已完成块仍然正常显示。
7. 并发度、超时时间、缓存策略具备可调空间。

## 10. 建议排除范围

为避免第一版过重，建议先不纳入以下范围：

- 图片中的文字识别翻译
- PDF/二进制附件内容翻译
- 用户输入中的实时翻译
- 多目标语言切换
- 自动整页无感翻译

第一版更适合做成“用户主动触发”的全局页内翻译。

## 11. 建议验收标准

满足以下条件即可认为这次需求完成：

1. 页面存在统一的全局悬浮翻译入口，而不是散落在单个 Markdown 组件里。
2. 系统可以识别当前页面是否存在英文内容。
3. 点击翻译后，可对当前页面中的英文文本块批量翻译。
4. 支持“替换原文”和“双语对照”两种展示模式。
5. 双语模式下，译文显示在原文下方，且页面结构不混乱。
6. 代码块、路径、命令、链接等非自然语言内容默认不翻译。
7. 页面重新渲染或切换模式时，已翻译内容不会无故丢失。
8. 已有译文命中缓存时，不重复调用翻译接口。

## 12. 建议拆分的实施阶段

### 阶段 1：能力抽象

- 把现有 `TranslateMarkdownView` 的局部逻辑抽出来
- 建立全局翻译状态管理
- 定义“可翻译块”的注册机制

### 阶段 2：全局悬浮面板

- 增加悬浮控制区
- 增加语言检测与全局操作入口
- 支持恢复原文

### 阶段 3：双语渲染

- 为每个内容块增加“原文 / 译文”的渲染能力
- 完成替换模式与对照模式切换

### 阶段 4：批量接口与缓存

- 后端新增批量翻译接口
- 前端升级块级缓存
- 优化失败重试与局部回退

## 13. 最终结论

你的需求不是“把现在的 md 翻译按钮再改一下”，而是要把现有的“文件级翻译”升级为“页面级翻译系统”。

核心变化有三点：

- 入口从局部按钮升级为全局悬浮控制区
- 粒度从单个 Markdown 文件升级为页面文本块
- 展示从原文/译文二选一，升级为“替换”与“双语对照”两种模式

如果后续要真正开始实现，建议优先做“全局状态层 + 块级注册”的方案，而不是直接在 DOM 上做一次性替换。
